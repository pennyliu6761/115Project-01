# 基於故障可預測性分析之銑床兩階段預測性維護決策支援系統

---

## 壹、研究問題與核心邏輯

**研究問題**：在銑床製程中，五種故障類型是否都適合用機器學習預測？如果不是，應該如何區分並採取不同策略？

**核心邏輯**：不同本質的故障需要不同的處理方式。傳統研究把所有故障混在一起跑模型，結果效能平平、也沒有洞見。本研究的突破點在於：先透過 EDA 分析故障的物理本質，再針對不同本質的故障設計對應的處理策略，最終形成一套兩階段決策支援系統。

---

## 貳、資料集說明

**資料集**：AI4I 2020 Predictive Maintenance Dataset，來源為 UCI Machine Learning Repository，可直接下載 CSV 檔。

**規模**：10,000 筆製程記錄，14 個欄位。

**五種故障的觸發條件**（整個研究的理論基礎，必須熟記）：

- **TWF（刀具磨耗故障）**：刀具磨耗時間達到 200–240 分鐘之間，「隨機」觸發。關鍵字是「隨機」，不是確定性的。
- **HDF（散熱不良故障）**：溫差（製程溫度 − 空氣溫度）< 8.6K，且轉速 < 1,380 rpm。確定性規則。
- **PWF（功率異常故障）**：轉速 × 扭矩（功率）< 3,500W 或 > 9,000W。確定性規則。
- **OSF（過負荷故障）**：功率 × 刀具磨耗超過各產品等級門檻（L 型 11,000、M 型 12,000、H 型 13,000 Nm·min）。確定性規則。
- **RNF（隨機故障）**：每次製程有 0.1% 的純隨機故障機率，與任何感測器值無關。完全隨機。

## 參、研究流程總覽

```
資料讀取與基本理解
        ↓
第一階段：EDA 與故障可診斷性分析
  ├─ 特徵工程（衍生 4 個新特徵）
  ├─ 各故障特徵分布視覺化
  ├─ 可診斷性量化實驗（1-vs-Rest AUC）
  ├─ TWF 貝葉斯誤差深度分析
  └─ 規則法 vs ML 基準比較
        ↓
  決策：RNF 排除建模、TWF 改用規則警示
        ↓
第二階段：ML / DL 模型比較（僅 HDF / PWF / OSF）
  ├─ 資料前處理（排除 TWF / RNF、SMOTE 平衡）
  ├─ 傳統 ML：Random Forest、XGBoost (Tuned)、LightGBM
  ├─ 深度學習：MLP (upgraded)、TabNet
  └─ 集成策略：Stacking (XGB + LGBM → RF)
        ↓
結果分析
  ├─ 三階段 MCC 進展比較（研究設計貢獻的核心展示）
  ├─ ML vs DL 差距分析
  ├─ 混淆矩陣分析
  └─ SHAP 可解釋性驗證（XGBoost vs TabNet）
        ↓
Streamlit UI 系統整合
```

## 肆、第一階段執行指引

### 步驟 1：資料讀取與基本探索

**做什麼**：讀入 CSV，了解資料結構、缺值情況、各欄位分布。

**程式方向**：
- 用 `pandas` 讀入，執行 `df.info()`、`df.describe()`
- 計算各故障類型的樣本數和比例
- 畫出各故障類別的樣本數長條圖

**預期輸出**：
- 確認 10,000 筆、無缺值
- 各故障樣本數：正常 8,183、TWF 770、HDF 83、PWF 925、OSF 297、RNF 9
- 正常樣本佔 81.8%，故障樣本僅 18.2%，類別嚴重不平衡

**分析重點**：這裡不只是確認數字，更要讓學生意識到「類別不平衡」這件事，它會影響後面所有的建模與評估決策，也是後續選擇 MCC 而非 Accuracy 作為主指標的原因。

---

### 步驟 2：特徵工程

**做什麼**：根據各故障的物理觸發條件，設計 4 個衍生特徵。

**具體計算**：
```python
df['Power'] = df['Rotational speed [rpm]'] * df['Torque [Nm]']
df['Power wear'] = df['Power'] * df['Tool wear [min]']
df['Temperature difference'] = df['Process temperature [K]'] - df['Air temperature [K]']

# 注意：Power = 0 時需要安全除法，否則會產生 inf 導致後續錯誤
df['Temperature power'] = np.where(
    df['Power'] != 0,
    df['Temperature difference'] / df['Power'],
    0.0
)
```

**為什麼這樣設計**：
- Power → 直接對應 PWF 的觸發條件（功率超出範圍）
- Power wear → 直接對應 OSF 的觸發條件（功率 × 磨耗超門檻）
- Temperature difference → 直接對應 HDF 的觸發條件（溫差過小）
- Temperature power → 複合特徵，後面消融實驗會驗證其貢獻

**注意**：這個特徵工程的設計理念很重要——特徵不是隨便組合，而是根據物理規則設計的。後面的 SHAP 分析也會反過來驗證這些特徵是否真的被模型重視。

---

### 步驟 3：各故障特徵分布視覺化

**做什麼**：針對每種故障，畫出該故障樣本與正常樣本在最相關特徵上的分布對比圖。

**建議畫的圖**：
- 刀具磨耗直方圖（TWF vs 正常）：用不同顏色疊加，並標示 200–250 分鐘的重疊區間
- 溫差分布圖（HDF vs 正常）：標示 8.6K 的規則邊界
- 功率分布圖（PWF vs 正常）：標示 3,500W 和 9,000W 的邊界
- 功率 × 磨耗分布圖（OSF vs 正常）：分 L/M/H 型別顯示不同門檻

**預期發現**：
- HDF、PWF、OSF 的圖會顯示故障樣本和正常樣本有清楚的分界，幾乎不重疊
- TWF 的圖會顯示在磨耗 200–250 分鐘區間，故障和正常樣本大量混雜在一起
- RNF 只有 9 筆，分布無規律，無法從圖中找到任何規律

**分析重點**：這一步是整個研究視覺上最有說服力的部分。學生要能從圖中直觀地說明「為什麼 HDF 和 PWF 容易預測，TWF 不容易預測」，讓讀者不看模型結果就能理解問題的本質差異。

---

### 步驟 4：故障可診斷性量化實驗（最重要的步驟）

**做什麼**：對每種故障分別做 1-vs-Rest 的二元分類實驗，用數字量化每種故障「有多可診斷」。

**方法**：
- 每次只取一種故障當正例，其餘所有資料（正常 + 其他故障）當負例
- 使用 XGBoost + 5-Fold StratifiedKFold 交叉驗證
- 每個 fold 中，以 scale_pos_weight 處理不平衡（= 負例數 / 正例數）
- 記錄 ROC-AUC、Average Precision（AP）、Recall

**為什麼用這個方法**：1-vs-Rest 可以獨立評估每種故障的可診斷性，排除其他故障相互干擾的影響。這是診斷「問題是否可解」最直接的方法。

**預期結果**：

| 故障 | 本質類型 | AUC | AP | Recall | 解讀 |
|---|---|---|---|---|---|
| HDF | 規則型 | 1.000 | 0.999 | 100% | 完全可診斷 |
| PWF | 規則型 | 1.000 | 0.999 | 99% | 完全可診斷 |
| OSF | 規則型 | 0.999 | 0.984 | 97% | 幾乎完全可診斷 |
| TWF | 半隨機型 | 0.935 | 0.398 | 95% | 有限預測力 |
| RNF | 純隨機型 | 0.503 | 0.003 | 0% | 完全無法預測 |

**分析重點**：這張表是第一階段最重要的輸出。重點不只是 AUC 的數字，還有 AP（Average Precision）的差距——TWF 的 AP 只有 0.398，代表在真正重要的「發現故障」這件事上，模型的 Precision 很低，誤報嚴重。

---

### 步驟 5：TWF 貝葉斯誤差深度分析

**做什麼**：解釋 TWF 為什麼有大量誤判，從資料本質找出根本原因。

**方法**：
- 統計磨耗 200–250 分鐘區間內，TWF 樣本和正常樣本各有多少筆
- 按每 10 分鐘一個區間，計算各區間的 TWF 發生率
- 比較被誤判為 TWF 的正常樣本和真正 TWF 樣本的感測器特徵均值

**預期發現**：
- 磨耗 200–250 區間共有 1,985 筆，其中 770 筆是 TWF，976 筆是正常製程
- 各 10 分鐘子區間的 TWF 發生率均約在 38–40%，沒有明顯遞增趨勢（純隨機）
- 被誤判的正常樣本與真正 TWF 的磨耗均值相差不到 0.1 分鐘，溫度、轉速、扭矩也幾乎完全相同

**結論的表達方式**：「在磨耗 200–250 分鐘區間，正常製程和 TWF 故障的所有感測器特徵分布完全相同，差別只在於 TWF 是否隨機觸發。在這種情況下，任何模型輸入的資訊都一樣，卻需要做不同的預測，這在資訊理論上是不可能的任務。這種誤差稱為貝葉斯誤差（Bayes Error），是資料本身的限制，無法透過改進模型來消除。」

**正確的工程決策**：不是「用更好的模型來預測 TWF」，而是「當磨耗達到 200 分鐘時，系統直接發出換刀警示」。這是規則型決策，Recall = 100%，邏輯透明，操作員容易理解和信任。

---

### 步驟 6：規則法 vs 機器學習基準比較

**做什麼**：用 if-else 規則直接套用物理公式做預測，和後面的 ML 結果對比，確認機器學習的必要性。

**規則設定**（直接翻譯物理條件）：
```python
rule_hdf = (temp_diff < 8.6) & (rpm < 1380)
rule_pwf = (power < 3500) | (power > 9000)
rule_osf = power_wear > threshold[product_type]  # L:11000 M:12000 H:13000
rule_twf = tool_wear >= 200
y_rule = rule_hdf | rule_pwf | rule_osf | rule_twf
```

**預期結果**：

| 方法 | Precision | Recall | F1 | MCC | 誤報筆數 |
|---|---|---|---|---|---|
| 規則法 | 18.1% | 100.0% | 0.306 | 0.000 | 8,192 |
| XGBoost (ML) | 64.5% | 97.0% | 0.775 | 0.737 | 193 |
| 提升幅度 | +46.4% | −3.0% | +0.469 | +0.737 | −97.6% |

**分析重點**：規則法雖然召回率 100%，但每 5 個警報有 4 個是假的。在工廠環境中，過多的假警報會導致操作員產生「警示疲勞（Alert Fatigue）」，開始忽略所有警示，反而比沒有系統更危險。機器學習的核心貢獻是將誤報降低 97.6%，同時只損失 3% 的召回率，這個取捨對工廠實務意義重大。

---

### 第一階段決策結論

根據步驟 4–6 的量化結果，得出兩個有數據支撐的設計決策：

**決策一**：RNF（AUC = 0.503，等同隨機猜測）不納入機器學習建模。含有 RNF 標籤的樣本在後續建模中將其 label 設為正常（Class 0），不單獨預測。

**決策二**：TWF 不用機器學習預測，改為規則型警示——「刀具磨耗達到 200 分鐘時，系統自動提醒管理者安排換刀評估，預防不可預測的異常發生」。

## 伍、第二階段執行指引

### 步驟 7：建立乾淨資料集

**做什麼**：根據第一階段決策，移除 TWF 和 RNF 樣本，建立用於機器學習的資料集。

**具體步驟**：
```python
# 排除 TWF 和 RNF 樣本
mask = (df['RNF'] == 0) & (df['TWF'] == 0)
df_clean = df[mask].copy()  # 剩餘約 9,221 筆

# 建立多元分類標籤（有重疊時以 HDF 優先，其次 OSF，再來 PWF）
df_clean['label'] = 0
df_clean.loc[df_clean['PWF'] == 1, 'label'] = 2
df_clean.loc[df_clean['OSF'] == 1, 'label'] = 3
df_clean.loc[df_clean['HDF'] == 1, 'label'] = 1  # 最後蓋掉，優先級最高
```

**重疊樣本說明**：有 88 筆樣本同時觸發多個故障，優先標記 HDF 的原因是散熱不良涉及整體機台溫度控制，比單純的功率或負荷問題更需要立即處理。

**確認資料分布**：
- 正常：8,183 筆（88.7%）
- HDF：75 筆（0.8%）
- PWF：767 筆（8.3%）
- OSF：196 筆（2.1%）

---

### 步驟 8：資料前處理

**做什麼**：完整的前處理流程，每一步的順序和理由都很重要。

**流程說明**：

**第一步，One-hot 編碼**：將 Type 欄位（L/M/H）轉為 Type_L 和 Type_M 兩個二元欄位，使用 drop_first=True 避免多重共線性。

**第二步，分割訓練集與測試集**：採 80/20 比例，使用 stratify=y 確保各類別比例一致，固定 random_state=0 確保結果可重現。

**第三步，特徵正規化**：MinMaxScaler 只在訓練集上 fit，再分別 transform 訓練集和測試集。絕對不能用全資料 fit，否則測試集的統計資訊洩漏到訓練過程（Data Leakage），評估結果會虛高。

**第四步，SMOTE 過採樣**：只在訓練集執行，k_neighbors=3（因為 HDF 只有約 60 筆訓練樣本，k 不能太大）。將四個類別均衡至各約 6,546 筆。測試集完全不做採樣，保持原始比例以正確反映真實環境的效能。

---

### 步驟 9：建立與訓練 6 個模型

**三個傳統 ML 模型（回答「傳統 ML 裡誰最好」）**：

**Random Forest**（n_estimators=100）：集成方法的代表，穩健性強，業界最常見的基準模型。

**XGBoost（調參版）**：以 RandomizedSearchCV 對 n_estimators、max_depth、learning_rate、subsample 等超參數做搜尋，以 MCC 為優化目標（使用 `make_scorer(matthews_corrcoef)`），n_iter=15，cv=3。記錄最佳參數組合。

**LightGBM**（n_estimators=100, max_depth=6, learning_rate=0.1）：比 XGBoost 速度更快的梯度提升框架，適合工廠生產環境的實際部署需求。

**兩個深度學習模型（回答「DL 和 ML 的差距在哪裡」）**：

**MLP（升級版）**（hidden_layer_sizes=(128,64,32), activation='relu', solver='adam', alpha=1e-4, learning_rate='adaptive', max_iter=300）：漸縮式網路架構，加入 L2 正則化（alpha）防止過擬合。不要使用 sample_weight，因為極少數類別的權重過高（HDF 約 109 倍）會導致訓練崩潰，改用 SMOTE 處理不平衡。

**TabNet**（n_d=16, n_a=16, n_steps=3, gamma=1.3, lambda_sparse=1e-3）：Google 提出的表格資料深度學習架構，使用稀疏注意力機制自動選擇每個決策步驟的重要特徵，可直接輸出全局特徵重要性（feature_importances_），是本研究中唯一能與 SHAP 做直接比較的深度學習模型。TabNet 需要 float32 格式的 numpy array 作為輸入，注意資料型別轉換。

**一個集成策略模型（回答「集成有沒有幫助」）**：

**Stacking（XGB + LGBM → RF）**：以 XGBoost 和 LightGBM 為 base learners，RandomForestClassifier（class_weight='balanced'）為 meta-learner，cv=3。注意 meta-learner 不要用 LogisticRegression，在多類別不平衡問題上 LR 容易收斂失敗，可能導致 Accuracy 崩潰至 30% 左右。

---

### 步驟 10：評估指標設定

**主要指標：MCC（Matthews Correlation Coefficient）**

為什麼選 MCC 而非 Accuracy？若將所有樣本預測為正常，Accuracy 仍有 88.7%，但這個模型對工廠毫無價值。MCC 同時考慮 TP、TN、FP、FN，對各類別不平衡最穩健，是預測性維護領域最推薦的評估指標。

**輔助指標**：
- ROC-AUC（multi_class='ovr', average='macro'）：各類別辨別能力的平均
- F1-Score（average='weighted'）：考慮類別支持度的加權 F1，注意必須明確指定 average 參數，否則預設 binary 會計算錯誤
- Recall（average='macro'）：各故障類別召回率的算術平均，反映對少數故障的偵測能力
- 訓練時間：用 `time.time()` 記錄，反映部署可行性

---

### 步驟 11：結果比較分析

**比較一：三階段 MCC 進展表（整個研究設計的核心貢獻展示）**

製作一張表，列出每個模型在三種設定下的 MCC：

| 模型 | 原始（含TWF/RNF） | 排除RNF | 排除TWF＆RNF（最終） | 總提升 |
|---|---|---|---|---|
| Random Forest | ~0.770 | ~0.758 | ~0.974 | +0.204 |
| XGBoost (Tuned) | ~0.782 | ~0.782 | ~0.968 | +0.186 |
| LightGBM | ~0.750 | ~0.759 | ~0.948 | +0.198 |
| MLP (upgraded) | ~0.547 | ~0.738 | ~0.951 | +0.404 |
| TabNet | ~0.441 | ~0.697 | ~0.941 | +0.500 |
| Stacking | ~0.773 | ~0.744 | ~0.968 | +0.195 |

表中數字是預期的大致範圍，實際執行後依結果填入。這張表的意義在於清楚說明：不是模型不夠好，而是問題定義不正確。當問題被正確定義後，所有模型的效能都大幅提升。

**比較二：ML vs DL 差距分析**

以 MCC 為 Y 軸、訓練時間為 X 軸（對數尺度），氣泡大小代表 AUC，畫氣泡圖。讓學生從圖中找出「效率效能最佳平衡點」在哪個模型，並解釋 DL 效能略低的四個原因：

第一，資料量約 9,200 筆，DL 通常需要十萬筆以上才能充分發揮。第二，特徵只有 11 個，邊界清晰，樹狀模型天生擅長低維切割問題。第三，HDF、PWF、OSF 的觸發條件都是數學規則，XGBoost 幾乎可以直接復現。第四，HDF 排除後只剩約 75 筆訓練樣本，DL 需要更多少數類別樣本才能學好邊界。

**比較三：混淆矩陣分析**

對每個模型畫混淆矩陣，分析三個問題：哪個故障類別最常被誤判？正常樣本最容易被誤判為哪種故障？HDF 的 Precision 和 Recall 是否明顯低於其他類別（因為樣本最少）？

## 柒、常見錯誤提醒

**一、在全資料上 fit Scaler**：MinMaxScaler 的 fit 只能用訓練集，否則測試集資訊洩漏，評估結果虛高。這是最常見、影響最嚴重的錯誤。

**二、SMOTE 在測試集上執行**：SMOTE 只能用在訓練集。測試集必須保持原始分布，否則評估結果不能反映真實環境的效能。

**三、F1-Score 沒有指定 average**：`f1_score()` 預設 binary，多分類問題必須加 `average='weighted'`，否則計算結果完全錯誤。

**四、Stacking 使用 LogisticRegression 當 meta-learner**：在多類別不平衡問題上，LR 容易收斂失敗，Accuracy 可能崩潰至 30% 以下。改用 RandomForestClassifier（class_weight='balanced'）。

**五、Temperature power 分母為零**：當 Power = 0 時，直接除法會產生 inf，後續 MinMaxScaler 會報錯。必須用 `np.where(Power != 0, Temp_diff / Power, 0.0)` 處理。

**六、XGBoost 不接受含括號的特徵名稱**：特徵名稱如 `Air temperature [K]` 中的 `[` 和 `]` 會造成 XGBoost 錯誤。訓練前將 DataFrame 轉為 numpy array，或用字典重新命名特徵。

**七、TabNet 記憶體限制**：TabNet 訓練時佔用較多記憶體，不要與其他大型模型在同一個 Python 程序中同時保留。訓練完後立即儲存結果指標和特徵重要性，釋放模型物件後再訓練下一個模型。
